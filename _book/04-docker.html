3  Docker

Autor: Łukasz Wawrowski

3.1 Wprowadzenie

Źródła wiedzy:

  • https://docs.docker.com/
  • https://www.freecodecamp.org/news/the-docker-handbook/
  • https://docker-curriculum.com/

Konteneryzacja jest jedną z najważniejszych technologii wykorzystywanych we współczesnych systemach chmurowych. Umożliwia ona uruchamianie aplikacji w izolowanych środowiskach zwanych kontenerami, które zawierają zarówno kod aplikacji, jak i wszystkie jej zależności. Dzięki temu aplikacje mogą być uruchamiane w sposób powtarzalny niezależnie od środowiska – na komputerze programisty, w centrum danych lub w chmurze publicznej.

W przeciwieństwie do tradycyjnych maszyn wirtualnych kontenery nie wymagają uruchamiania pełnego systemu operacyjnego. Zamiast tego wykorzystują mechanizmy izolacji systemu operacyjnego hosta, dzięki czemu są znacznie lżejsze i uruchamiają się szybciej. Takie podejście pozwala efektywniej wykorzystywać zasoby sprzętowe oraz ułatwia skalowanie aplikacji w środowiskach chmurowych.

Jedną z najpopularniejszych platform konteneryzacji jest Docker. Platforma ta składa się z kilku podstawowych elementów. Najważniejszym z nich jest demon Dockera, czyli proces działający w tle, który odpowiada za budowanie, uruchamianie i zarządzanie kontenerami. Użytkownik komunikuje się z nim za pomocą klienta Docker CLI, czyli narzędzia wiersza poleceń. Obrazy kontenerów przechowywane są zazwyczaj w rejestrach, z których najpopularniejszym jest Docker Hub.

3.2 Instalacja

Wszystkie możliwości instalacji dockera w systemie ubuntu są przedstawione na stronie. Poniżej instrukcja dla metody używającej repozytorium apt.

W pierwszej kolejności podajemy dane repozytorium dockera:

# Add Docker's official GPG key:
sudo apt update
sudo apt install ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

# Add the repository to Apt sources:
sudo tee /etc/apt/sources.list.d/docker.sources <<EOF
Types: deb
URIs: https://download.docker.com/linux/ubuntu
Suites: $(. /etc/os-release && echo "${UBUNTU_CODENAME:-$VERSION_CODENAME}")
Components: stable
Signed-By: /etc/apt/keyrings/docker.asc
EOF

sudo apt update

A następnie instalujemy najnowszą wersję narzędzi:

sudo apt install docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

Możemy zweryfikować czy usługi dockera zostały automatycznie uruchomione:

sudo systemctl status docker

Jeśli tak się nie stało to możemy je wystartować:

sudo systemctl start docker

Spróbuj znaleźć w sieci informacje o tym, jak firma Google wykorzystała kontenery do optymalizacji swoich centrów danych.

3.3 Podstawowe pojęcia w ekosystemie Docker

Podstawowym elementem pracy z Dockerem jest obraz kontenera. Obraz można traktować jako szablon aplikacji, zawierający wszystkie niezbędne pliki, biblioteki oraz konfigurację. Na podstawie obrazu można uruchamiać wiele instancji kontenerów, które są jego działającymi egzemplarzami. Obrazy mogą być pobierane z publicznych rejestrów lub budowane samodzielnie przez użytkownika. Przykładowo można uruchomić aplikację w dwóch różnych wersjach.

Kontener jest uruchomioną instancją obrazu. W praktyce oznacza to izolowany proces działający w systemie operacyjnym hosta. Kontener posiada własny system plików, interfejsy sieciowe oraz konfigurację środowiska, jednak współdzieli jądro systemu operacyjnego z hostem. Takie rozwiązanie zapewnia wysoką wydajność przy jednoczesnym zachowaniu izolacji aplikacji.

Aby uruchomić kontener, wykorzystywana jest komenda docker run. Polecenie to powoduje utworzenie nowego kontenera na podstawie wskazanego obrazu oraz uruchomienie określonego procesu w jego wnętrzu. Jeżeli obraz nie jest dostępny lokalnie, Docker automatycznie pobiera go z rejestru. Dzięki temu wdrażanie aplikacji może być zrealizowane za pomocą pojedynczej komendy.

$ docker run hello-world

Hello from Docker.
This message shows that your installation appears to be working correctly.
...

3.4 Pobieranie obrazu z docker hub

Zanim zbudujemy własny obraz możemy skorzystać z wielu dostępnych w repozytorium dockera. Do pobierania służy komenda:

docker pull busybox

Po pobraniu obraz zapisywany jest na komputerze i gotowy do użycia:

docker images

3.4.1 Zadanie

Pobierz z repozytorium Docker Hub oficjalny obraz systemu Ubuntu.

3.5 Uruchamianie kontenerów i podstawowe polecenia

Jeśli kontener zostanie uruchomiony bez dodatkowych parametrów, wykona on swoje zadanie i natychmiast zakończy pracę. Aby wejść w interakcję z wnętrzem kontenera, można użyć flag -it, które dołączają interaktywny terminal użytkownika do procesu wewnątrz kontenera.

Zarządzanie cyklem życia kontenerów obejmuje również ich monitorowanie i usuwanie. Polecenie ps pozwala sprawdzić aktualnie działające jednostki, a flaga -a rozszerza ten widok o kontenery, które już zakończyły pracę. Ważną praktyką jest regularne czyszczenie systemu z nieużywanych kontenerów za pomocą komendy rm lub narzędzi do automatycznego porządkowania, co zapobiega niepotrzebnemu zużyciu miejsca na dysku.

docker run -it busybox sh

3.5.1 Zadanie

Uruchom kontener Busybox z flagą --rm, wykonaj w nim komendę ls i sprawdź po wyjściu z niego, czy kontener nadal widnieje na liście docker ps -a.

3.6 Wdrażanie aplikacji webowych i mapowanie portów

Docker znacząco upraszcza uruchamianie serwerów WWW, pozwalając na ich szybkie odpalenie z gotowych obrazów dostępnych w rejestrach. Podczas pracy z usługami sieciowymi kluczowe jest zrozumienie trybu odłączonego (detached mode), który pozwala na działanie kontenera w tle, nawet po zamknięciu okna terminala. W takim trybie konieczne jest jednak odpowiednie opublikowanie portów, aby host mógł komunikować się z aplikacją zamkniętą w kontenerze.

Użytkownik ma do wyboru automatyczne mapowanie portów na losowe numery lub ręczne przypisanie konkretnego portu maszyny fizycznej do portu, na którym nasłuchuje aplikacja wewnątrz kontenera. Dzięki temu można łatwo uruchomić wiele instancji tej samej aplikacji na jednym serwerze, kierując ruch na różne porty zewnętrzne.

Pobierz obraz https://hub.docker.com/r/prakhar1989/static-site/ i uruchom go.

docker run -p 8888:80 prakhar1989/static-site

3.6.1 Zadanie

Uruchom kontener ze stroną statyczną prakhar1989/static-site w trybie odłączonym (-d) i nadaj mu nazwę własną moja-strona.

3.7 Tworzenie własnego obrazu aplikacji

Proces tworzenia niestandardowego obrazu opiera się na pliku tekstowym o nazwie Dockerfile, który zawiera zestaw instrukcji automatyzujących budowę środowiska. Każda linia w tym pliku odpowiada za utworzenie nowej warstwy obrazu, co pozwala na optymalizację i ponowne wykorzystanie gotowych elementów. Instrukcje w Dockerfile są bardzo zbliżone do standardowych komend systemowych Linux, dzięki czemu ich tworzenie jest intuicyjne dla osób znających terminal. Poprawnie sformułowany plik pozwala na spakowanie aplikacji wraz ze wszystkimi jej zależnościami w jedną, gotową do uruchomienia jednostkę.

Struktura typowego pliku Dockerfile zaczyna się od instrukcji FROM, która definiuje obraz bazowy, stanowiący fundament dla naszych modyfikacji. Następnie, za pomocą komendy WORKDIR, określa się katalog roboczy wewnątrz kontenera, do którego kopiowane są pliki aplikacji przy użyciu instrukcji COPY lub ADD. Kluczowym elementem jest instalacja niezbędnych pakietów i bibliotek poprzez instrukcję RUN oraz wskazanie portów komunikacyjnych za pomocą EXPOSE. Na końcu pliku umieszcza się instrukcję CMD, która definiuje domyślne polecenie wykonywane automatycznie w momencie startu kontenera.

Gotowy projekt zamienia się w obraz za pomocą polecenia docker build, w którym flaga -t (tag) pozwala nadać obrazowi czytelną nazwę i wersję. Kropka na końcu komendy wskazuje na kontekst budowy, czyli bieżący katalog, w którym Docker szuka pliku Dockerfile i plików aplikacji. Po pomyślnym zbudowaniu obrazu można go uruchomić jako kontener za pomocą komendy docker run, pamiętając o mapowaniu portów, aby aplikacja była dostępna z poziomu hosta.

Zrozumienie różnicy między restartem kontenera a budową nowego obrazu jest kluczowe dla efektywnej pracy. Nowy obraz należy zbudować zawsze wtedy, gdy zmienia się „receptura” środowiska, czyli instrukcje w Dockerfile (np. dodanie nowej biblioteki systemowej) lub gdy zmieniamy kod aplikacji, który został na stałe wbudowany (skopiowany) do obrazu podczas procesu budowy. Natomiast restart kontenera wystarczy w sytuacjach, gdy potrzebujemy jedynie ponownego uruchomienia procesów wewnątrz już istniejącego środowiska, na przykład po awarii aplikacji.

Prosty Dockerfile dla statycznej strony opartej na Nginx:

FROM ubuntu:latest
RUN apt-get update && apt-get install nginx -y
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Budowanie i uruchomienie:

docker build -t moj-nginx:v1 .
docker run -d -p 8080:80 moj-nginx:v1

3.7.1 Zadanie:

Stwórz Dockerfile dla prostej aplikacji w wybranym języku (np. Python lub Node.js), zbuduj obraz o nazwie moja-aplikacja:v1 i uruchom go. Następnie zmień tekst wyświetlany przez aplikację w pliku źródłowym i sprawdź, czy zwykły restart kontenera (docker restart) wystarczy, aby zmiany były widoczne, czy konieczne jest ponowne zbudowanie obrazu.

3.8 Środowiska wielokontenerowe

W praktycznych zastosowaniach aplikacje rzadko składają się z jednego kontenera. Typowa aplikacja internetowa może obejmować kilka komponentów, takich jak serwer aplikacji, baza danych, system cache czy wyszukiwarka. Zarządzanie taką architekturą przy użyciu pojedynczych poleceń docker run byłoby skomplikowane i podatne na błędy.

Rozwiązaniem tego problemu jest narzędzie Docker Compose. Jest to narzędzie umożliwiające definiowanie i uruchamianie środowisk wielokontenerowych przy użyciu jednego pliku konfiguracyjnego. W pliku tym opisuje się wszystkie usługi wchodzące w skład aplikacji oraz relacje pomiędzy nimi.

Docker Compose wywodzi się z projektu o nazwie Fig, który został stworzony w celu ułatwienia pracy z wieloma kontenerami w środowiskach programistycznych. Popularność tego narzędzia doprowadziła do jego przejęcia przez firmę Docker i przekształcenia w oficjalny projekt Docker Compose. Obecnie jest on standardowym narzędziem do zarządzania aplikacjami wielokontenerowymi w wielu środowiskach – od lokalnego developmentu po środowiska testowe i systemy CI/CD.

3.9 Plik docker-compose.yml

Konfiguracja środowiska wielokontenerowego zapisywana jest w pliku docker-compose.yml. Plik ten wykorzystuje składnię YAML i zawiera definicję wszystkich usług wchodzących w skład aplikacji. Każda usługa odpowiada jednemu kontenerowi oraz określa obraz, porty, wolumeny danych oraz inne parametry uruchomienia.

Przykładowa konfiguracja może zawierać dwie usługi: serwer aplikacji oraz usługę wyszukiwania opartą na Elasticsearch. Dla każdej z nich określany jest obraz kontenera, a także dodatkowe parametry konfiguracyjne, takie jak zmienne środowiskowe czy mapowanie portów.

Istotnym elementem konfiguracji są wolumeny danych. Pozwalają one na przechowywanie danych poza systemem plików kontenera, dzięki czemu dane mogą być zachowane nawet po usunięciu kontenera. Jest to szczególnie ważne w przypadku baz danych lub systemów przechowujących duże ilości informacji.

Plik konfiguracyjny może również definiować zależności pomiędzy usługami. Parametr depends_on pozwala określić, że jeden kontener powinien zostać uruchomiony dopiero po starcie innego. Dzięki temu można zapewnić poprawną kolejność uruchamiania elementów systemu, na przykład najpierw bazy danych, a następnie aplikacji korzystającej z tej bazy.

3.10 Uruchamianie środowiska z Docker Compose

Po przygotowaniu pliku konfiguracyjnego środowisko wielokontenerowe może zostać uruchomione za pomocą pojedynczej komendy docker compose up. Polecenie to automatycznie pobiera wymagane obrazy, tworzy kontenery oraz konfiguruje sieć pomiędzy nimi. W efekcie wszystkie komponenty aplikacji uruchamiane są jednocześnie i mogą komunikować się ze sobą w ramach wspólnej sieci.

Docker Compose automatycznie tworzy również sieć wirtualną dla wszystkich usług zdefiniowanych w pliku konfiguracyjnym. Dzięki temu kontenery mogą odnajdywać się wzajemnie przy użyciu nazw usług jako nazw hostów. Mechanizm ten znacznie upraszcza konfigurację komunikacji pomiędzy komponentami aplikacji.

Środowisko może zostać uruchomione w trybie interaktywnym lub w trybie odłączonym. W trybie odłączonym (-d) kontenery działają w tle, a użytkownik odzyskuje dostęp do terminala. Jest to szczególnie przydatne podczas pracy z aplikacjami webowymi lub w środowiskach serwerowych.

3.11 Zarządzanie cyklem życia środowiska

Docker Compose umożliwia również łatwe zarządzanie cyklem życia aplikacji. Polecenie docker compose ps pozwala wyświetlić listę uruchomionych usług oraz informacje o ich stanie i portach. W ten sposób można szybko zweryfikować, czy wszystkie komponenty aplikacji działają poprawnie.

Zatrzymanie środowiska odbywa się za pomocą polecenia docker compose down. Komenda ta zatrzymuje i usuwa wszystkie kontenery utworzone przez Compose oraz usuwa utworzoną sieć. Opcjonalnie możliwe jest również usunięcie wolumenów danych, co powoduje całkowite usunięcie wszystkich danych aplikacji.

Dzięki zastosowaniu Docker Compose zarządzanie złożonymi aplikacjami składającymi się z wielu usług staje się znacznie prostsze. Cała konfiguracja infrastruktury aplikacji zapisana jest w jednym pliku, co ułatwia jej wersjonowanie, automatyzację wdrożeń oraz integrację z narzędziami ciągłej integracji i dostarczania oprogramowania.

3.11.1 Zadanie

Pobierz i uruchom przykładowy zestaw kontenerów ze strony https://github.com/docker/awesome-compose/tree/master/wordpress-mysql

3.12 Zarządzanie wolumenami i trwałość danych

Kontenery Dockera są z natury efemeryczne, co oznacza, że wszelkie dane utworzone lub zmodyfikowane wewnątrz nich zostają bezpowrotnie utracone w momencie usunięcia kontenera. Aby zapobiec utracie istotnych informacji, takich jak wpisy w bazach danych czy pliki przesłane przez użytkowników, stosuje się wolumeny. Są one mechanizmem pozwalającym na przechowywanie danych poza cyklem życia samego kontenera, na systemie plików maszyny hosta, co umożliwia ich zachowanie nawet po zatrzymaniu i skasowaniu instancji aplikacji.

Zarządzanie wolumenami odbywa się za pomocą dedykowanych poleceń CLI, które pozwalają na ich tworzenie, wyświetlanie oraz usuwanie. Docker oferuje różne typy trwałości danych, z których najczęściej stosowane to wolumeny nazwane (zarządzane całkowicie przez Dockera) oraz montowania typu bind (umożliwiające bezpośrednie powiązanie katalogu z hosta z katalogiem w kontenerze). Stosowanie nazwanych wolumenów jest szczególnie polecane w scenariuszach produkcyjnych, gdyż Docker dba o ich strukturę i optymalizację, a użytkownik może łatwo odwoływać się do nich po prostej nazwie podczas uruchamiania kolejnych usług.

Tworzenie nazwanego wolumenu i podpięcie go do bazy danych PostgreSQL:

docker volume create dane-bazy 
docker run -d --name moja-baza -v dane-bazy:/var/lib/postgresql/data postgres:12

3.12.1 Zadanie

Utwórz nazwany wolumen o nazwie moje-pliki, a następnie uruchom kontener z obrazem linux alpine w trybie interaktywnym (-it), montując ten wolumen w katalogu /dane. Wewnątrz kontenera stwórz plik testowy poleceniem touch /dane/test.txt, wyjdź z kontenera, usuń go i sprawdź, czy po uruchomieniu nowego kontenera z tym samym wolumenem plik test.txt nadal istnieje w katalogu /dane.
